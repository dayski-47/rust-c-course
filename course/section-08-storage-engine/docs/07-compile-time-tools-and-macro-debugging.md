# Doc 07 — Compile-Time Tools and Macro Debugging

🟡 Rust compiles slower than C. The difference between a painful development loop and a fast one is using the right tools. This doc covers the tools that cut compile times, give you visibility into what the compiler is doing, and make macro errors debuggable.

---

## Why Rust Compiles Slowly

Rust's front-loaded safety model moves work into the compiler that other languages defer to runtime. The borrow checker, trait resolution, const evaluation, monomorphization, and LTO all add to compile time. Understanding where time goes helps you address the right bottleneck.

```bash
# See what's taking time per crate
cargo build --timings

# Opens a flame chart in your browser showing:
# - Which crates take the most time
# - Codegen vs frontend time
# - Whether you're benefiting from parallelism
```

The two biggest levers on compile time:

1. **Fewer dependencies** — every dependency compiles. 200 dependencies means 200 compilation units.
2. **Link time** — the final linking step is serial and often the bottleneck on incremental builds.

## Faster Linking with `mold`

The default GNU `ld` linker is slow. On Linux, `mold` is 3–10× faster for the linking step:

```bash
# Install on Ubuntu 22.04+
sudo apt install mold

# Or from source: https://github.com/rui314/mold
```

```toml
# .cargo/config.toml
[target.x86_64-unknown-linux-gnu]
rustflags = ["-C", "link-arg=-fuse-ld=mold"]
```

The impact on ironkv: each edit-compile-test cycle goes from ~8 seconds to ~3 seconds on the link step alone. On a project with many crates, the savings compound.

On macOS, the built-in linker is already fast. On Windows, use the `lld` linker (part of LLVM) as the mold alternative.

## Compilation Caching with `sccache`

`sccache` wraps `rustc` and caches compilation outputs. When you clean the build directory or switch branches, cached crates don't need to recompile:

```bash
cargo install sccache

# Configure as the Rust compiler wrapper
export RUSTC_WRAPPER=sccache

# Or permanently in .cargo/config.toml:
# [build]
# rustc-wrapper = "sccache"

# First build: normal speed (populates cache)
cargo build --release  # 3 minutes

# Clean + rebuild: cache hits for unchanged crates
cargo clean && cargo build --release  # ~45 seconds

# Check cache statistics
sccache --show-stats
```

`sccache` supports team-shared caches via S3, GCS, or Azure Blob. In CI, configure a shared cache so the first run of a new branch benefits from the cache built by main branch runs.

## `cargo-nextest`: A Better Test Runner

`cargo-nextest` runs tests in parallel per-test rather than per-file, and provides much better failure output:

```bash
cargo install cargo-nextest

# Run tests (significantly faster on multi-core machines)
cargo nextest run --workspace

# Re-run only failed tests
cargo nextest run --workspace --no-fail-fast

# With verbose output for debugging
cargo nextest run -- test_name_filter
```

Compared to `cargo test`:
- Runs each test in its own process — prevents test-to-test interference
- Shows a live progress bar with per-test timing
- Failure output is grouped and readable, not interleaved
- On a 16-core machine, can be 2–5× faster than `cargo test`

One caveat: `cargo nextest` can't run doctests. Add `cargo test --workspace --doc` as a separate CI step.

## `cargo-expand`: Seeing What Macros Generate

Proc macros, derive macros, and complex `macro_rules!` generate code you can't see in your editor. `cargo-expand` shows the expanded output:

```bash
cargo install cargo-expand

# Expand all macros in the crate root
cargo expand

# Expand a specific module
cargo expand storage

# Expand with pretty-printed output
cargo expand --color always | less -R
```

For ironkv, after writing `#[derive(Storable)]`, run `cargo expand` to see exactly what code was generated. A bug in the macro shows up as wrong code in the expanded output — usually obvious to fix once you can see it.

Example of what you'll see when things go wrong:

```bash
$ cargo expand storage::index

# Generated by #[derive(Storable)] for IndexEntry:
impl Storable for IndexEntry {
    fn to_bytes(&self) -> Vec<u8> {
        let mut bytes = Vec::new();
        let field_bytes = self.key.to_storable_bytes();  // ← this is what you generated
        bytes.extend_from_slice(&(field_bytes.len() as u64).to_le_bytes());
        bytes.extend_from_slice(&field_bytes);
        // ... offset, length fields ...
        bytes
    }
    // ...
}
```

If `to_storable_bytes()` is the wrong method name, you see it immediately in the expansion.

## `cargo-watch`: Continuous Compilation

During development, you want the compiler running continuously — catch errors as you type:

```bash
cargo install cargo-watch

# Recompile on any source change
cargo watch -x build

# Recompile and run tests
cargo watch -x test

# Recompile, then run specific test
cargo watch -x "test put_and_get"

# Multiple commands
cargo watch -x check -x test -x run
```

For ironkv specifically, a good development loop:

```bash
# Terminal 1: watch for compile errors
cargo watch -x "check --workspace"

# Terminal 2: run tests on save
cargo watch -x "nextest run --workspace"
```

This catches compilation errors in ~1 second (just cargo check, no codegen) and test failures in ~3 seconds.

## Workspace Lints: Uniform Quality Across All Crates

Instead of adding lint configuration to every crate's `lib.rs`, configure lints once in the workspace `Cargo.toml`:

```toml
# Cargo.toml (workspace root)
[workspace.lints.rust]
unsafe_code = "warn"          # Warn on unsafe (we use it carefully in ironkv)
unused_imports = "deny"       # Hard error on unused imports
unused_variables = "warn"     # Warn on unused variables

[workspace.lints.clippy]
all = { level = "warn", priority = -1 }      # All default Clippy lints as warnings
pedantic = { level = "warn", priority = -1 } # Pedantic lints (stricter)
# Allow specific lints that produce false positives in storage code:
cast_possible_truncation = "allow"  # We know our byte lengths fit in u32
```

Each member crate inherits these lints:

```toml
# ironkv-core/Cargo.toml
[lints]
workspace = true  # Inherit from workspace
```

Individual crates can override specific lints when there's a legitimate reason:

```rust
// In ironkv-core/src/unsafe_io.rs — we use unsafe deliberately here
#![allow(unsafe_code)]  // Override the workspace "warn" for this file

// Safety comment required per project policy:
// Safety: The mmap pointer is valid for the lifetime of the MmapFile struct.
unsafe { ... }
```

## Compile-Time Feature Verification with `cargo-hack`

Feature flags can create combinations that don't compile. `cargo-hack` catches them systematically:

```bash
cargo install cargo-hack

# Verify each feature compiles in isolation
cargo hack check --each-feature --workspace

# Verify all features together
cargo check --workspace --all-features

# Verify with no default features
cargo check --workspace --no-default-features
```

For ironkv's features:

```toml
# ironkv-core/Cargo.toml
[features]
default = []
compression = ["dep:zstd"]    # Optional zstd compression
encryption = ["dep:aes-gcm"]  # Optional AES-GCM encryption
cli = ["dep:clap"]            # CLI utilities
```

`cargo hack check --each-feature` catches the case where `encryption` accidentally depends on `compression` being enabled (because you used a function only available with both features). Without `cargo-hack`, this bug hides until a user reports "it doesn't compile with just `encryption`."

Add to CI:

```yaml
- name: Feature matrix check
  run: cargo hack check --each-feature --workspace --no-dev-deps
```

## MSRV: Minimum Supported Rust Version

Document and enforce the minimum Rust version your crate supports:

```toml
# ironkv-core/Cargo.toml
[package]
rust-version = "1.75"  # MSRV: must be stable, specify exactly
```

Verify that the MSRV compiles:

```yaml
# .github/workflows/msrv.yml
jobs:
  msrv:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@1.75
      - run: cargo check --workspace
```

For a library like ironkv-core, a stable MSRV matters if users embed it in projects with fixed toolchain versions. For application binaries like ironkv-cli, you can track stable more aggressively.

## Documentation as CI

Docs with broken links or examples that don't compile are worse than no docs. Run documentation checks in CI:

```yaml
- name: Check documentation
  run: RUSTDOCFLAGS='-Dwarnings' cargo doc --workspace --no-deps
```

`-Dwarnings` treats doc warnings (missing documentation, broken links) as errors. Combined with `--no-deps` (don't try to document dependencies), this checks ironkv's own documentation without noise from third-party crates.

For ironkv's public API, every `pub` function, struct, and trait should have at least a one-line doc comment:

```rust
/// Open a storage engine at the given path, creating it if it doesn't exist.
///
/// # Errors
///
/// Returns an error if the path is not writable, if the existing data file
/// is corrupt, or if the WAL cannot be replayed.
pub fn open(path: impl AsRef<std::path::Path>) -> Result<Self, Error> {
    // ...
}
```

The `# Errors` section is especially important — it tells users what they need to handle with `?` or `match`.

## The Iterative Compile Loop

The goal: fast feedback from every code change. The ideal loop:

1. **Cargo check** (< 2s): catches 90% of errors without codegen
2. **Cargo nextest** (< 5s for unit tests): catches behavioral regressions
3. **Cargo build** (< 10s for incremental): needed before running the actual binary
4. **Full CI** (< 5 min): catches cross-platform and cross-feature issues

Tools that enable this:
- `cargo-watch` for automatic check on save
- `mold` for fast linking
- `sccache` for caching unchanged dependencies
- `cargo-nextest` for parallel test execution
- `cargo check` instead of `cargo build` during active development
