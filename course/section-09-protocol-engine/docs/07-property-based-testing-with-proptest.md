# Doc 07 - Property-Based Testing with Proptest

🟡 Unit tests check specific inputs. Property-based tests check that a property
holds for *all* inputs - where "all" means a large random sample generated by the
testing framework. For nexus, the most valuable properties are: frames that encode
always decode, the type-state machine never allows invalid transitions, and the
topic routing table always routes to the correct subscribers.

---

## What Property-Based Testing Is

Traditional tests:

```rust
#[test]
fn test_frame_parse_small_payload() {
    let frame = Frame { command: Command::Ping, payload: Bytes::new(), sequence_id: 0, ... };
    let encoded = encode_frame(&frame);
    let decoded = decode_frame(&encoded).unwrap();
    assert_eq!(frame, decoded);
}
```

This tests one specific case. You probably also test an empty payload, a large payload,
a payload with null bytes. But you're choosing which cases to test.

Property-based test:

```rust
proptest! {
    #[test]
    fn encoded_frame_always_decodes(
        command in valid_command(),
        payload in arbitrary_payload(0..=1024),
        sequence_id in 0u64..=u64::MAX,
    ) {
        let frame = Frame { command, payload, sequence_id, version: 1 };
        let encoded = encode_frame(&frame);
        let decoded = decode_frame(&encoded).unwrap();
        prop_assert_eq!(frame, decoded);
    }
}
```

This runs hundreds of iterations with random values. The framework finds inputs that
break the property and reports the minimal counterexample.

---

## How Proptest Works: Generate, Fail, Shrink

Proptest runs tests in three phases:

**1. Generate**: Proptest uses a strategy (a description of what to generate) to produce
random inputs. For `any::<u64>()`, it generates random 64-bit integers. For your custom
`valid_frame()`, it composes strategies to produce a complete `Frame`. It runs the test
body with each generated input.

**2. Fail**: If the test body panics or `prop_assert!` fails with any input, proptest
records that input as a failure seed.

**3. Shrink**: Proptest tries smaller or simpler values derived from the failing input.
For a string strategy, it tries shorter strings. For a vec strategy, it tries fewer
elements. For a numeric strategy, it tries smaller numbers. This continues until no
simpler value still fails. The result is the *minimal* counterexample - often just 1-3
elements that reproduce the bug.

Shrinking is what makes proptest genuinely useful over random testing. A raw fuzzer
gives you a 50,000-byte input that crashes the decoder. Proptest gives you a 3-byte
sequence that triggers the same bug. The minimal case immediately shows *what the
invariant violation actually is*.

## Writing Strategies

A *strategy* generates random values. Proptest has built-in strategies for primitives;
you compose them for your types:

```rust
// nexus-proto/tests/prop_codec.rs
use proptest::prelude::*;
use bytes::Bytes;
use crate::{Frame, Command};

// Strategy for valid command bytes
fn valid_command() -> impl Strategy<Value = Command> {
    prop_oneof![
        Just(Command::Connect),
        Just(Command::Publish),
        Just(Command::Subscribe),
        Just(Command::Unsubscribe),
        Just(Command::Ping),
        Just(Command::Disconnect),
    ]
}

// Strategy for arbitrary byte payloads up to max_bytes
fn arbitrary_payload(max_bytes: usize) -> impl Strategy<Value = Bytes> {
    prop::collection::vec(any::<u8>(), 0..=max_bytes)
        .prop_map(Bytes::from)
}

// Strategy for topic names: ASCII alphanumeric with `:` and `/` separators
fn valid_topic() -> impl Strategy<Value = String> {
    // Use a regex strategy for structured strings
    "[a-z][a-z0-9_/:.]{0,63}".prop_map(|s| s)
}

// Strategy for a complete valid frame
fn valid_frame() -> impl Strategy<Value = Frame> {
    (valid_command(), arbitrary_payload(1024), any::<u64>()).prop_map(
        |(command, payload, sequence_id)| Frame {
            version: 1,
            command,
            payload_len: payload.len() as u32,
            sequence_id,
            payload,
        }
    )
}
```

## Core Properties to Test

### 1. Codec Roundtrip

The most fundamental property: anything that encodes must decode to the same value.

```rust
proptest! {
    #[test]
    fn frame_encode_decode_roundtrip(frame in valid_frame()) {
        let mut buf = bytes::BytesMut::new();
        let mut codec = NexusCodec::new(2 * 1024 * 1024);  // 2 MB limit

        // Encode
        codec.encode(frame.clone(), &mut buf).unwrap();

        // Decode
        let decoded = codec.decode(&mut buf).unwrap()
            .expect("Should have a complete frame");

        prop_assert_eq!(frame.version, decoded.version);
        prop_assert_eq!(frame.command, decoded.command);
        prop_assert_eq!(frame.sequence_id, decoded.sequence_id);
        prop_assert_eq!(frame.payload, decoded.payload);
    }
}
```

### 2. Partial Frame Returns None

The decoder must return `None` (not `Err`) when data is incomplete:

```rust
proptest! {
    #[test]
    fn partial_frame_returns_none(frame in valid_frame(), truncate_at in 0usize..=15usize) {
        let mut buf = bytes::BytesMut::new();
        let mut codec = NexusCodec::new(1024 * 1024);

        codec.encode(frame, &mut buf).unwrap();

        // Truncate to a partial frame
        let partial: bytes::BytesMut = buf[..truncate_at].into();

        // A partial frame must return None, not an error
        let result = codec.decode(&mut partial.clone()).unwrap();
        prop_assert!(result.is_none(), "Partial frame ({truncate_at} bytes) should return None");
    }
}
```

### 3. Arbitrary Bytes Don't Panic

The decoder must not panic on any input - even malformed or adversarial bytes.
It should return `Err`, not panic.

```rust
proptest! {
    #[test]
    fn arbitrary_bytes_never_panic(bytes in prop::collection::vec(any::<u8>(), 16..=1024)) {
        let mut buf = bytes::BytesMut::from(bytes.as_slice());
        let mut codec = NexusCodec::new(1024 * 1024);

        // May succeed, fail, or return None - but must not panic
        let _ = codec.decode(&mut buf);
    }
}
```

This is the parser fuzzing property. Run this test and you're doing property-based
fuzzing. Run it with `cargo +nightly fuzz` and you're doing coverage-guided fuzzing.

### 4. Message Routing Consistency

For any sequence of subscribes and publishes, the routing table must be consistent:

```rust
use proptest::collection::vec;

#[derive(Debug, Clone)]
enum TopicOp {
    Subscribe { conn_id: u64, topic: String },
    Unsubscribe { conn_id: u64, topic: String },
    Publish { topic: String, seq: u64 },
}

fn topic_op() -> impl Strategy<Value = TopicOp> {
    prop_oneof![
        (0u64..100, valid_topic()).prop_map(|(id, t)| TopicOp::Subscribe { conn_id: id, topic: t }),
        (0u64..100, valid_topic()).prop_map(|(id, t)| TopicOp::Unsubscribe { conn_id: id, topic: t }),
        (valid_topic(), 0u64..10000).prop_map(|(t, s)| TopicOp::Publish { topic: t, seq: s }),
    ]
}

proptest! {
    #[test]
    fn topic_routing_is_consistent(ops in vec(topic_op(), 1..=50)) {
        use std::collections::{HashMap, HashSet};

        let mut subscriptions: HashMap<String, HashSet<u64>> = HashMap::new();
        let mut deliveries: Vec<(String, u64, u64)> = Vec::new();  // (topic, seq, conn_id)

        // Reference model: a simple HashMap
        for op in &ops {
            match op {
                TopicOp::Subscribe { conn_id, topic } => {
                    subscriptions.entry(topic.clone()).or_default().insert(*conn_id);
                }
                TopicOp::Unsubscribe { conn_id, topic } => {
                    if let Some(subs) = subscriptions.get_mut(topic) {
                        subs.remove(conn_id);
                    }
                }
                TopicOp::Publish { topic, seq } => {
                    if let Some(subs) = subscriptions.get(topic) {
                        for conn_id in subs {
                            deliveries.push((topic.clone(), *seq, *conn_id));
                        }
                    }
                }
            }
        }

        // Apply same ops to nexus TopicMap
        let topic_map = crate::topic::TopicMap::new();
        let mut actual_deliveries: Vec<(String, u64, u64)> = Vec::new();

        for op in ops {
            match op {
                TopicOp::Subscribe { conn_id, topic } => {
                    let (_, rx) = tokio::sync::mpsc::channel(16);
                    topic_map.subscribe(&topic, conn_id, rx);
                }
                TopicOp::Unsubscribe { conn_id, topic } => {
                    topic_map.unsubscribe(&topic, conn_id);
                }
                TopicOp::Publish { topic, seq } => {
                    let delivered = topic_map.route(&topic, seq, bytes::Bytes::new());
                    for conn_id in delivered {
                        actual_deliveries.push((topic.clone(), seq, conn_id));
                    }
                }
            }
        }

        // Delivery sets must match
        let expected_set: HashSet<_> = deliveries.iter().collect();
        let actual_set: HashSet<_> = actual_deliveries.iter().collect();
        prop_assert_eq!(expected_set, actual_set);
    }
}
```

This tests a property: *the topic routing table always routes exactly like a
reference HashMap implementation*. Any deviation is a bug in the routing logic.

## Shrinking: Finding the Minimal Counterexample

When proptest finds a failing test, it automatically *shrinks* the input - finding
the smallest input that still causes the failure. This is proptest's killer feature.

Without shrinking, a failing input might be:
```
ops: [Subscribe{conn_id=47, topic="metrics/cpu"}, Publish{topic="alerts", seq=1}, 
      Subscribe{conn_id=3, topic="alerts"}, Unsubscribe{conn_id=47, topic="metrics/cpu"},
      Publish{topic="alerts", seq=2}, Subscribe{conn_id=99, topic="metrics/cpu"},
      Publish{topic="alerts", seq=3}]
```

After shrinking:
```
ops: [Subscribe{conn_id=0, topic="a"}, Publish{topic="a", seq=0}]
```

The bug: publishing to a topic where the subscriber list was just created has an
off-by-one. The shrunk case makes this immediately obvious.

## Configuration

By default, proptest runs 256 cases per test. Adjust for your situation:

```rust
proptest! {
    // Run 1000 cases for security-critical parsers
    #![proptest_config(ProptestConfig::with_cases(1000))]

    #[test]
    fn arbitrary_bytes_never_panic(bytes in prop::collection::vec(any::<u8>(), 16..=1024)) {
        let mut buf = bytes::BytesMut::from(bytes.as_slice());
        let _ = NexusCodec::new(1024 * 1024).decode(&mut buf);
    }
}
```

In CI, proptest saves failing seeds to a `proptest-regressions/` directory. Commit
this directory. Failed tests from CI reproduce locally:

```bash
# Run with saved regression seed
PROPTEST_REGRESSIONS=proptest-regressions cargo test
```

```toml
# Cargo.toml - don't exclude the regression directory
[package.metadata.proptest]
failure_persistence = "direct"
```

## State Machine Testing

For the nexus connection state machine, test that transitions are always valid:

```rust
#[derive(Debug, Clone)]
enum ConnectionAction {
    SendConnect { token: String },
    SendPublish { topic: String, payload: Vec<u8> },
    SendSubscribe { topic: String },
    SendDisconnect,
}

fn connection_action() -> impl Strategy<Value = ConnectionAction> {
    prop_oneof![
        "[a-zA-Z0-9]{8,32}".prop_map(|t| ConnectionAction::SendConnect { token: t }),
        (valid_topic(), prop::collection::vec(any::<u8>(), 0..=64))
            .prop_map(|(t, p)| ConnectionAction::SendPublish { topic: t, payload: p }),
        valid_topic().prop_map(|t| ConnectionAction::SendSubscribe { topic: t }),
        Just(ConnectionAction::SendDisconnect),
    ]
}

proptest! {
    #[test]
    fn connection_state_machine_is_consistent(
        actions in prop::collection::vec(connection_action(), 1..=20)
    ) {
        // Reference model: track expected state
        let mut state = "Unauthenticated";
        let valid_auth_token = "correct-token";

        for action in &actions {
            match (state, action) {
                ("Unauthenticated", ConnectionAction::SendConnect { token }) => {
                    if token == valid_auth_token {
                        state = "Active";
                    }
                    // stays Unauthenticated on bad token
                }
                ("Active", ConnectionAction::SendDisconnect) => {
                    state = "Closed";
                }
                ("Active", ConnectionAction::SendPublish { .. }) => {}  // OK
                ("Active", ConnectionAction::SendSubscribe { .. }) => {}  // OK
                ("Unauthenticated", ConnectionAction::SendPublish { .. }) => {
                    // This should return an auth error, not panic
                }
                ("Closed", _) => {
                    // All operations on Closed should return an error
                }
                _ => {}
            }
        }
        // Property: we never reached an invalid state
        prop_assert!(["Unauthenticated", "Active", "Closed"].contains(&state));
    }
}
```

## Integration with Criterion Benchmarks

Proptest and Criterion complement each other. After proptest finds an edge case,
benchmark it with Criterion:

```rust
use criterion::{black_box, criterion_group, criterion_main, Criterion};
use proptest::prelude::*;

// First: proptest found that large topics with many subscribers are slow.
// Now: benchmark to measure HOW slow.

fn bench_routing_many_subscribers(c: &mut Criterion) {
    let topic_map = TopicMap::new();
    
    // Set up 1000 subscribers to "alerts"
    let mut receivers = Vec::new();
    for id in 0..1000u64 {
        let (_, rx) = tokio::sync::mpsc::channel(128);
        topic_map.subscribe("alerts", id, rx);
        receivers.push(rx);
    }

    let payload = bytes::Bytes::from(vec![0u8; 128]);

    c.bench_function("route_1000_subscribers", |b| {
        b.iter(|| {
            topic_map.route("alerts", black_box(1), payload.clone());
        });
    });
}
```

This is the testing pyramid for nexus: proptest generates the interesting cases,
Criterion measures their performance.

## Running Property Tests

```bash
# Run all tests including proptest
cargo test

# Run specifically the property tests, verbose output
cargo test prop_

# Run with more cases (override default 256)
PROPTEST_CASES=1000 cargo test

# Run with a specific seed for reproduction
PROPTEST_SEED=12345 cargo test

# Generate an HTML report (using proptest's test runner)
cargo test -- --nocapture 2>&1 | grep proptest
```

In CI, add a separate step to run property tests with more iterations:

```yaml
# .github/workflows/ci.yml
- name: Property tests (extended)
  env:
    PROPTEST_CASES: 1000
  run: cargo test --test prop_
```

## What to Test with Proptest

**High value:**
- Codec roundtrips (encode → decode = original)
- Parser robustness (arbitrary bytes never panic)
- Routing consistency (matches reference HashMap model)
- Sequence number monotonicity (always increases under concurrent load)

**Lower value (use unit tests instead):**
- Specific error messages (exact string matching is fragile)
- Specific protocol sequences (better as integration tests)
- Authentication logic (small, discrete - just enumerate cases)
